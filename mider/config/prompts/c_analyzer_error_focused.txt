## 분석 절차 (Chain-of-Thought)

### Step 1: 정적분석 경고 전수 분석 (필수)
**위 정적분석 결과의 모든 경고를 빠짐없이 분석하세요.**
경고가 10개이면 최소 10개 이슈가 있어야 합니다. 특히:
- UNINIT_VAR (초기화 없는 변수): 해당 변수가 이후 어디서 사용되는지 추적하여 반드시 이슈로 보고
- UNSAFE_FUNC (위험 함수): strcpy, sprintf 등 경계 미검증 함수는 반드시 이슈로 보고
- MALLOC_NO_CHECK: malloc/calloc 반환값 NULL 미검증은 반드시 이슈로 보고

각 경고에 대해:
1. 경고가 발생한 코드 라인을 확인합니다
2. 실제 장애 가능성을 판단합니다
3. 수정 방안을 제시합니다

### Step 2: 심층 패턴 분석
clang-tidy가 감지하지 못하는 아래 패턴을 추가로 검사합니다:

**버퍼 오버플로우 (memory_safety)**
- strcpy, sprintf 사용 (strncpy, snprintf 대신)
- 배열 크기를 넘는 인덱스 접근
- gets() 사용 (fgets 대신)

**NULL 포인터 역참조 (null_safety)**
- malloc/calloc 반환값 NULL 체크 누락
- 함수 반환 포인터 NULL 체크 누락
- 포인터 매개변수 NULL 검증 누락

**메모리 누수 (memory_safety)**
- malloc 후 free 누락 (특히 에러 경로)
- 함수 중간 return 시 할당된 메모리 미해제
- realloc 실패 시 원본 포인터 유실

**Use-After-Free / Double-Free (memory_safety)**
- free 후 포인터 재사용
- 같은 포인터를 두 번 free
- free 후 포인터를 NULL로 설정하지 않음

**스레드 안전성 (performance)**
- 전역 변수 접근 시 mutex 미사용
- 공유 자원 접근 시 동기화 누락

### Step 3: 이슈 정리
- source가 "static_analysis"인 이슈: clang-tidy가 탐지한 것
- source가 "llm"인 이슈: LLM이 추가로 발견한 것
- source가 "hybrid"인 이슈: clang-tidy 결과를 LLM이 재분석한 것

## 이슈 예시
{{
  "issue_id": "C-001",
  "category": "memory_safety",
  "severity": "critical",
  "title": "strcpy 사용으로 버퍼 오버플로우 위험",
  "description": "strcpy는 대상 버퍼의 크기를 검증하지 않아 입력 문자열이 버퍼보다 크면 인접 메모리를 덮어씁니다. 이는 프로그램 크래시 또는 원격 코드 실행으로 이어질 수 있습니다.",
  "location": {{
    "file": "/app/service/calc.c",
    "line_start": 23,
    "line_end": 23,
    "column_start": 5,
    "column_end": 30
  }},
  "fix": {{
    "before": "strcpy(dest, src);",
    "after": "strncpy(dest, src, sizeof(dest) - 1);\ndest[sizeof(dest) - 1] = '\\0';",
    "description": "strncpy를 사용하여 복사 길이를 제한하고 null 종료를 보장합니다."
  }},
  "source": "hybrid",
  "static_tool": "clang-tidy",
  "static_rule": "bugprone-not-null-terminated-result"
}}

{{

      "severity": "critical|high|medium|low",
      "title": "한국어 제목",
      "description": "한국어 상세 설명",
      "location": {{
        "file": "{file_path}",
        "line_start": 0,
        "line_end": 0,
        "column_start": null,
        "column_end": null
      }},
      "fix": {{
        "before": "수정 전 코드 (1-3줄)",
        "after": "수정 후 코드 (1-3줄)",
        "description": "한국어 수정 설명"
      }},
      "source": "static_analysis|llm|hybrid",
      "static_tool": "clang-tidy 또는 null",
      "static_rule": "규칙명 또는 null"
    }}
  ]
}}

## 주의사항
- 이슈 ID는 C-001부터 순차적으로 부여합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다.
- 발견된 이슈가 없으면 `"issues": []`로 응답합니다.
- C 언어의 메모리 안전성 이슈는 severity를 높게 책정합니다 (critical 또는 high).
- **정적분석 결과의 모든 경고를 누락 없이 이슈로 변환하세요. 경고를 건너뛰지 마세요.**
