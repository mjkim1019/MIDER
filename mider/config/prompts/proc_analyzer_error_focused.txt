## 분석 절차 (Chain-of-Thought)

### Step 1: proc 에러 분석
각 proc 에러에 대해:
1. 에러가 발생한 EXEC SQL 구문을 확인합니다
2. 에러의 실제 영향도를 판단합니다
3. 수정 방안을 제시합니다

### Step 2: SQLCA 에러 처리 검사 (data_integrity)
각 EXEC SQL 블록에 대해:
1. 실행 후 `sqlca.sqlcode` 또는 `WHENEVER SQLERROR` 검사가 있는지 확인합니다
2. DML 문(INSERT, UPDATE, DELETE)은 반드시 에러 체크가 필요합니다
3. SELECT INTO는 "데이터 없음"(sqlcode == 1403) 처리가 필요합니다

### Step 3: INDICATOR 변수 검사 (null_safety)
각 SELECT/FETCH 구문에서:
1. NULL 가능 컬럼에 INDICATOR 변수가 있는지 확인합니다
2. INDICATOR 변수가 없으면 NULL 값 수신 시 프로그램이 비정상 동작할 수 있습니다

### Step 4: 커서 라이프사이클 검사 (data_integrity)
1. DECLARE CURSOR에 대응하는 OPEN/CLOSE가 있는지 확인합니다
2. 에러 경로에서도 커서가 닫히는지 확인합니다
3. CLOSE 없이 프로그램이 종료되면 DB 자원 누수가 발생합니다

### Step 5: 트랜잭션 무결성 검사 (data_integrity)
1. EXEC SQL COMMIT/ROLLBACK의 위치가 적절한지 확인합니다
2. 에러 발생 시 ROLLBACK이 실행되는지 확인합니다
3. COMMIT 후 에러 발생 시 부분 커밋(partial commit) 문제가 없는지 확인합니다

### Step 6: Heuristic Scanner 결과 검증 (장애 유발 패턴)
Scanner가 탐지한 패턴을 실제 코드에서 확인합니다:

**FORMAT_STRUCT** — %s 포맷에 구조체 전달 (Core Dump 유발)
- 위험: `PFM_DSP("값=[%s]", io.in.arr[0]);` → 구조체 전체가 %s에 전달 → Core Dump
- 정상: `PFM_DSP("값=[%s]", io.in.arr[0].field);` → 문자열 멤버 접근

**MEMSET_SIZEOF_MISMATCH** — memset 변수/sizeof 타입 불일치 (이전 데이터 잔류)
- 위험: `memset(&xxx_u0010_in, 0, sizeof(xxx_s0009_in_t));` → 크기 다를 수 있음
- 정상: `memset(&xxx_u0010_in, 0, sizeof(xxx_u0010_in));` → 변수 자체 크기 사용

**LOOP_INIT_MISSING** — 루프 내 초기화 누락 (데이터 누적)
- 위험: while루프에서 INIT2VCHAR가 주석 처리됨 → 이전 반복 데이터가 잔류
- 정상: 매 반복 시작 시 INIT2VCHAR 또는 memset으로 구조체 초기화

**FCLOSE_MISSING** — fopen/fclose 짝 불일치 (파일 핸들 릭)
- 위험: fopen 후 fclose 없음 → 장시간 배치 시 "Too many open files"

## 주의사항
- 이슈 ID는 PC-001부터 순차적으로 부여합니다.
- Pro*C는 데이터 무결성이 가장 중요합니다. SQLCA 미검사, 커서 미닫힘은 severity를 높게 책정합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다.
- 발견된 이슈가 없으면 `"issues": []`로 응답합니다.
- **[Proframe 환경] 아래 패턴은 이슈로 보고하지 마세요:**
  - INDICATOR 변수 누락 — Proframe은 WHENEVER SQLERROR + NVL()로 NULL을 처리하므로 INDICATOR 불필요
  - memset에서 sizeof(타입명) 사용 — Proframe 표준 관행이며 타입명과 변수 타입이 일치
  - 전역 변수 동시성/스레드 안전성 — Proframe은 프로세스 기반 단일스레드
  - NULL 포인터 검증 누락 — INPUT/NGMHEADER/ctx는 프레임워크가 보장
- **[중요] fix.before/after에는 원본 코드에 실제로 존재하는 변수명만 사용하세요.**
