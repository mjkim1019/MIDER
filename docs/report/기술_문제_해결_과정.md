# 기술 문제 해결 과정

테스트 및 고도화 과정에서 발견된 기술적 문제들의 상황, 원인, 해결 방법, 결과를 기록합니다.

---

### 1. Proframe/레거시 특화 프롬프트 정합성 문제

**문제 상황**: 일반 Pro*C 프롬프트가 Proframe 특유 매크로/함수 패턴(seq_open/seq_close 등)을 인식 못해 오탐/누락 발생

**원인**: 도메인/프레임워크 특화 규칙이 프롬프트에 포함되지 않아 LLM이 일반화된 가정으로 판단

**해결 방법**: proc_analyzer 프롬프트에 Proframe 환경 특성 및 핵심 패턴을 명시적으로 추가

**결과**: 레거시 환경에서 의미 있는 이슈 설명/패치 제안 품질이 개선됨

---

### 2. LLM 패치 출력 포맷 깨짐(Before/After)

**문제 상황**: LLM이 생성한 Before/After 코드에 \n 리터럴이 남거나, 중괄호 블록이 한 줄로 뭉쳐 리포트 가독성이 저하

**원인**: LLM 출력의 escape/format 변형 및 코드 블록 정규화 로직 부재

**해결 방법**: `_expand_code_lines()`로 \n 리터럴을 줄바꿈으로 변환하고 `{ stmt1; stmt2; }` 패턴을 줄단위로 정렬

**결과**: 터미널 패널 및 JSON 리포트에서 패치 제안의 재현성/가독성 향상

---

### 3. clang-tidy 헤더 의존성으로 분석 중단

**문제 상황**: 폐쇄망/단건 파일 분석 환경에서 프로젝트 헤더 부재로 clang-tidy가 `#include` fatal error → Level 2 분석 불가

**원인**: 빌드 컨텍스트(헤더/컴파일 옵션)가 없는 상태에서 정적분석 도구가 파싱을 중단

**해결 방법**: `StubHeaderGenerator`로 include 목록을 파싱해 임시 stub 헤더(공통 타입/커스텀 타입 확장)를 자동 생성하고 `-I stubs/`로 전달

**결과**: clang-tidy 분석 성공률 및 데이터 흐름 경고 탐지 범위가 확대되고, 분석 후 stubs 자동 삭제로 환경 오염 방지

---

### 4. Explain Plan 핵심 패턴 누락(CARTESIAN JOIN 등)

**문제 상황**: LLM이 Explain Plan의 치명적 성능 패턴(CARTESIAN JOIN 등)을 누락하는 케이스 발생

**원인**: LLM 기반 해석의 변동성으로 특정 패턴을 안정적으로 커버하지 못함

**해결 방법**: Explain Plan 파서 결과(tuning_points)에서 규칙 기반 정적 이슈를 생성하고 LLM 이슈와 병합(source=hybrid)

**결과**: 치명 패턴 탐지의 안정성(재현성) 향상

---

### 5. 대용량 C 파일 분석 시간 증가(정확도-속도 트레이드오프)

**문제 상황**: 2-pass(선별→심층) 도입 후 대용량 파일에서 분석 시간이 2분 목표를 초과

**원인**: 함수별 gpt-5 호출 횟수 증가로 지연 누적

**해결 방법**: (현재) 토큰 최적화(구조 요약+함수 윈도우)로 비용을 먼저 절감, (향후) 비동기 병렬 분석/캐시 도입 계획 수립

**결과**: 탐지율은 개선(단일 패스 대비 10→14건)됐으나, 처리 시간은 추가 최적화 필요

---

### 6. Few-Shot 실제 장애 사례 프롬프트로 위험 함수 선별 정확도 80% 향상

**문제 상황**: 2-Pass 전략의 Pass 1(gpt-5-mini)에서 위험 함수를 선별할 때, 일반적인 지시 프롬프트만으로는 LLM이 오탐(안전한 함수를 위험으로 판정)과 누락(실제 위험 함수를 건너뜀)을 반복. 특히 ProFrame 환경 특유의 패턴(선언 직후 대입하는 `long rc;` 패턴, memset의 변수명-타입명 숫자 일치 규칙 등)을 LLM이 일반적인 C 코드 관점으로 잘못 판단하여 false positive가 과다 발생하고, 정작 `curcnt` 미초기화 배열 인덱스나 `%s`에 long형 전달 같은 실제 장애 유발 패턴은 누락

**원인**: 프롬프트에 도메인 특화 판단 기준이 없어 LLM이 일반적인 C 코드 규칙으로 판정. ProFrame 코드에서는 `long rc;` 후 즉시 `rc = pfm_init(ctx);` 대입이 표준 패턴이고, memset의 변수명(u0010)과 sizeof 타입명(s0009)의 숫자 불일치가 핵심 위험 신호이지만, 이런 도메인 지식을 LLM이 갖고 있지 않음. 또한 stub 헤더 환경에서 발생하는 미정의 타입/암시적 함수 선언 등 환경 아티팩트를 실제 위험으로 오판

**해결 방법**: `c_prescan_fewshot.txt` 프롬프트에 실제 운영 장애 사례 기반 Few-Shot 예시 5건(위험 3건 + 안전 2건)을 구성하여 LLM의 판단 기준을 명시적으로 교정

| 구분 | 예시 함수 | 패턴 | 판정 근거 |
|------|----------|------|----------|
| 위험 1 | `c400_get_svc_list` | `long curcnt, ix;` 미초기화 → 배열 인덱스/루프 제어에 쓰레기 값 | 실제 장애: 루프 미실행으로 데이터 누락 |
| 위험 2 | `c100_abn_sale_proc` | `memset(&u0010, 0x00, sizeof(s0009_t))` 변수-타입 숫자 불일치 | 실제 장애: 부분 초기화로 이전 트랜잭션 데이터 잔류 |
| 위험 3 | `c200_auto_chrg_proc` | `PFM_ERRS("%s", ctx->svc_mgmt_num)` long형에 %s 포맷 | 실제 장애: Core Dump 발생 |
| 안전 1 | `a000_init_proc` | `long rc;` 직후 `rc = pfm_init(ctx);` 대입 | ProFrame 표준 패턴, 실제 위험 없음 |
| 안전 2 | `z999_err_exit_proc` | `memset(&ctx, 0x00, sizeof(ctx_t))` 숫자 일치 | 크기 보장, 위험 예시 2와 대비하여 판별 |

추가로 stub 헤더 환경에서 발생하는 오류 5종(미정의 타입, 암시적 함수 선언, 헤더 미발견, 축소 변환, strncpy LEN_* 안전)을 **제외 대상**으로 명시하여 환경 아티팩트 오탐을 차단

**결과**:
- 위험 함수 선별 정확도 **~80% 향상** (오탐 대폭 감소 + 실제 장애 패턴 탐지 재현성 확보)
- Pass 2에 불필요한 함수가 전달되지 않아 gpt-5 호출 횟수 감소 → **비용 효율 동시 개선**
- 안전 예시를 통해 ProFrame 도메인 특화 판단이 가능해져 false positive 억제
