# Issue #003: Pass 2 대형 함수 지배로 소형 함수 분석 누락

|             |                                      |                                                                                           |
| ----------- | ------------------------------------ | ----------------------------------------------------------------------------------------- |
| **이슈 구분**   | **문제 상황 및 원인**                       | **리서치 및 해결 과정 (Reference & Solution)**                                                    |
| **Problem** | 2-Pass 분석에서 Pass 1이 위험 함수 4개(c100, c200, c400, c700)를 올바르게 선별했으나, Pass 2에서 4개 함수를 **한 번의 LLM 호출**에 묶어 전달하면 대형 함수(c100+c200)만 분석하고 소형 함수(c400, c700)를 누락함 | - **원인:** 4개 함수 합산 2042줄 + 65개 grouped warnings를 한 프롬프트에 전달하면, LLM이 앞쪽 대형 함수를 처리하다 output token을 소진하거나 attention이 분산되어 뒤쪽 소형 함수를 건너뜀 |
| **증상** | Pass 2 결과에서 c100(5개 이슈) + c200(5개 이슈) = 10개만 반환. c400의 `svc_cnt` 미초기화(UNINIT_VAR)와 c700의 이슈는 완전히 누락 | - **확인:** 프롬프트에 "모든 함수를 빠짐없이 분석하세요", "첫 번째 함수만 분석하고 끝내지 마세요" 등 지시를 추가했으나 효과 없음. 함수별 그룹화 warnings도 시도했으나 동일하게 누락 |
| **근본 원인** | 단일 LLM 호출에 2042줄의 코드를 전달하는 구조적 문제. LLM의 attention window 한계로, 입력이 길수록 뒤쪽 내용의 처리 품질이 떨어짐 (lost-in-the-middle 현상) | - **임시 조치:** `MIDER_EXCLUDE_FUNCTIONS` 환경변수로 대형 함수 제외 가능 (테스트/디버그용) |

## 현재 분석 흐름 (문제)

```
Pass 2: 위험 함수 4개를 한 번에 LLM에 전달

┌──────┬────────┬──────────┬───────────────┐
│ 함수 │  코드  │ warnings │     결과      │
├──────┼────────┼──────────┼───────────────┤
│ c100 │ 636줄  │ 116개    │ 이슈 5개 생성 │
├──────┼────────┼──────────┼───────────────┤
│ c200 │ 1115줄 │ 226개    │ 이슈 5개 생성 │
├──────┼────────┼──────────┼───────────────┤
│ c400 │ 127줄  │ 20개     │ 누락          │
├──────┼────────┼──────────┼───────────────┤
│ c700 │ 164줄  │ 49개     │ 누락          │
└──────┴────────┴──────────┴───────────────┘

문제: c100+c200 = 1751줄이 프롬프트를 압도 → c400/c700 무시
```

## 시도한 해결책 (실패)

1. **프롬프트 강화**: "모든 함수를 빠짐없이 분석하세요" 지시 추가 → 효과 없음
2. **함수별 그룹화 warnings**: HIGH 우선순위 패턴 먼저 배치, 함수당 최대 15개 제한 → 효과 없음
3. **MIDER_EXCLUDE_FUNCTIONS**: 대형 함수를 환경변수로 제외 → 임시 workaround (c400 이슈 탐지 성공하지만 c100/c200 분석 포기)

---

## 해결 방안: 함수별 개별 LLM 호출

### 변경 후 흐름

```
Pass 2: 함수별 개별 LLM 호출

c100 (636줄, 116 warnings) → gpt-4o 호출 #1 → 이슈 N개
c200 (1115줄, 226 warnings) → gpt-4o 호출 #2 → 이슈 N개
c400 (127줄, 20 warnings)  → gpt-4o 호출 #3 → 이슈 N개
c700 (164줄, 49 warnings)  → gpt-4o 호출 #4 → 이슈 N개

→ 4개 결과 합산 → AnalysisResult
```

### 장점

| 항목 | Before (일괄) | After (개별) |
|------|--------------|-------------|
| c400 svc_cnt 탐지 | X (누락) | O |
| LLM attention 분산 | 심각 (2042줄) | 없음 (함수 단위) |
| 분석 일관성 | 앞쪽 함수만 상세 | 모든 함수 균등 |
| LLM 호출 횟수 | 1회 | 4회 (함수 수만큼) |
| 비용 | 1회 × 대형 | N회 × 소형 (총량 유사) |

### 구현 포인트

1. `_run_two_pass()` 내부에서 `risky_functions`를 for-loop으로 순회
2. 각 함수별로 코드 추출 + warnings 생성 + LLM 호출
3. 결과를 합산하여 issues 리스트 반환
4. `MIDER_EXCLUDE_FUNCTIONS` workaround 제거
5. `asyncio.gather()`로 병렬 호출 가능 (비용 동일, 시간 단축)

### 비용 분석

- 기존: 1회 호출 × 2042줄 입력 = ~2000 input tokens
- 변경: 4회 호출 × (636+1115+127+164)줄 입력 = ~2000 input tokens (동일)
- output은 함수당 별도이므로 오히려 품질 향상
