# 2단계: 상세 설계 및 개발 환경 구축

## Agent 상세 설계

### Agent 아키텍처 개요

```
┌─────────────────────────────────────────────────┐
│                  CLI (main.py)                    │
│            --files --explain-plan --output        │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│            OrchestratorAgent                     │
│  Phase 0 → Phase 1 → Phase 2 → Phase 3           │
└──┬────────┬────────┬────────┬────────┬──────────┘
   │        │        │        │        │
   ▼        ▼        ▼        ▼        ▼
 Task    Context   JS/C/   ProC/XML  Reporter
Classifier Collector SQL    Analyzer  Agent
 Agent     Agent   Analyzer  Agent
```

### Agent 1: OrchestratorAgent

#### 1. Agent 페르소나 및 시스템 프롬프트 (Identity)

| 항목 | 정의 내용 |
| ---- | -------- |
| **Agent 이름** | OrchestratorAgent |
| **주요 역할** | 전체 분석 워크플로우 제어 및 Sub-agent 조율 총괄. Phase 0 → 1 → 2 → 3 순차 실행하며 각 단계의 입출력을 관리 |
| **핵심 목표** | - 사용자가 지정한 파일들을 분석하여 배포 차단 이슈를 탐지<br>- 각 Phase의 결과를 통합하여 최종 리포트 생성 |
| **톤앤매너** | 명령형, 시스템 관리자 스타일. 간결하고 정확한 상태 보고 |
| **제약 사항** | - 코드를 직접 수정하지 않음 (제안만)<br>- 사용자가 선택한 파일만 분석 (프로젝트 전체 탐색 금지)<br>- Phase 순서 고정 (0 → 1 → 2 → 3) |

#### 2. 워크플로우 및 오케스트레이션 (Workflow & Logic)

##### 2.1 처리 로직

**Step 1 (Input Analysis)**

```text
1. 파일 경로 검증
   - 존재 여부 확인
   - 읽기 권한 확인
   - 와일드카드 확장 (GlobTool)
   - 지원 확장자 확인 (.js, .c, .h, .pc, .sql, .xml)
2. 중복 제거 (Path.resolve() 기반)
3. TaskClassifierAgent 호출 준비
```

**Step 2 (Phase Orchestration)**

```text
Phase 0: Task Classification
  → TaskClassifierAgent.run(files=valid_files)
  → ExecutionPlan 획득 (sub_tasks, dependencies)
  → Progress 콜백 보고

Phase 1: Context Collection
  → ContextCollectorAgent.run(execution_plan)
  → FileContext 획득 (imports, calls, patterns)
  → context_map 생성 (파일별 매핑)

Phase 2: Sequential Language Analysis
  For each task in ExecutionPlan.sub_tasks:
    → 언어별 AnalyzerAgent 인스턴스 캐싱 (같은 언어 재사용)
    → _analyze_single_file() 호출
    → SQL 분석 시 explain_plan_file 전달
    → AnalysisResult 수집

Phase 3: Report Generation
  → ReporterAgent.run(analysis_results, session_id, ...)
  → IssueList, Checklist, Summary, DeploymentChecklist 생성
```

**Step 3 (Execution & Response)**

```text
1. 터미널 출력 (Rich Console)
   - 진행률 표시 (Phase별 done/시간)
   - Phase 2는 파일별 진행률 [1/N]
   - Critical/High 이슈 Before/After 코드 패널
   - 심각도별 요약 바
   - 배포 가능 여부 판정
2. 파일 출력 (JSON)
   - ./output/issue-list.json
   - ./output/checklist.json
   - ./output/summary.json
   - ./output/deployment-checklist.json
```

##### 2.2 상태 관리

* **Stateless**: 세션 저장/복구 없음
* **session_id**: UUID 기반 (uuid4().hex[:12])

#### 3. 도구(Tools) 및 함수 명세 (Capability)

| 도구명 | 기능 설명 | 입력 파라미터 | 출력 데이터 |
| ------ | -------- | ----------- | --------- |
| **GlobTool** | 와일드카드 파일 패턴 확장 | pattern: str, root: str | matched_files: List[str] |
| **FileReader** | 파일 내용 읽기 | path: str | content: str |
| **_validate_files** | 파일 존재 및 권한/확장자 검증 | filepaths: List[str] | valid_files, errors |
| **_call_agent** | Sub-agent run() 호출 | agent: BaseAgent, **kwargs | agent_result: Dict |

#### 4. 지식 베이스 및 메모리 전략 (Context & Memory)

* **메모리 유형**: None (세션 저장 없음)

#### 5. 핵심 에이전트 기술 스택

| 구분 | 선정 전략/기술 | 선정 사유 |
| ---- | ------------ | -------- |
| **LLM Model** | gpt-5 (Fallback: gpt-5-mini) | 심층 분석용, Azure OpenAI 배포 |
| **Agent Framework** | Python Class (BaseAgent ABC) | 워크플로우가 명확하고 순차적이므로 복잡한 프레임워크 불필요 |

| **핵심 목표** | - 파일 확장자 기반 언어 식별 (JS/C/ProC/SQL/XML)<br>- import/include 기반 의존성 그래프 생성<br>- Critical 가능성 높은 파일 우선순위 부여<br>- 분석 순서 최적화 (의존성 역순) |

#### 2. 워크플로우 및 오케스트레이션 (Workflow & Logic)

##### 2.1 처리 로직

**Step 1 (Language Detection):**

```text
For each file in selected_files:
  1. 확장자 추출
     - .js → JavaScript
     - .c, .h → C
     - .pc → Pro*C
     - .sql → SQL
     - .xml → XML (WebSquare/Proframe)
  2. 파일 메타데이터 수집
     - 파일 크기
     - 라인 수
     - 수정 날짜
```

**Step 2 (Dependency Analysis):**

```text
1. 파일별 import/include 추출
   JavaScript: import, require 구문
   C: #include 지시문
   Pro*C: EXEC SQL INCLUDE, #include
   SQL: N/A (의존성 없음)
   XML: 이벤트 핸들러 참조 (ev:onclick 등)
2. 의존성 그래프 생성
   파일 A → 파일 B (A가 B를 참조)
3. 순환 의존성 검사
   발견 시 경고 메시지
```

**Step 3 (Priority & Order):**

```text
1. 우선순위 기준
   - 의존성: 하위 계층 먼저 (SQL → C → JS)
   - 복잡도: 라인 수 많을수록 높음
   - 수정 날짜: 최근 수정 파일 우선
2. 실행 순서 결정 (Topological Sort)
3. ExecutionPlan 생성
```

##### 2.2 상태 관리

* **Stateless**: 입력(파일 리스트) → 출력(ExecutionPlan)만 생성


**Step 3 (Pattern Analysis):**

```text
공통 패턴 탐지:
  - 에러 처리: try-catch, if-return, SQLCA check
  - 로깅: console.log, printf, log_error
  - 트랜잭션: COMMIT, ROLLBACK
  - 메모리 관리: malloc/free 쌍
→ FileContext 생성
```

##### 2.1 처리 로직

**Step 1 (Static Analysis - ESLint):**

```text
eslint_runner(file_path, config=".eslintrc.json")
→ errors: [{"rule": "no-undef", "message": "...", "line": 45}, ...]
```

**Step 2 (LLM Deep Analysis — 토큰 최적화: Structure + Function Window):**

```text
경로 A (ESLint errors found — Error-Focused):
  structure_summary = _build_structure_summary(file_context)
    → imports, 함수 시그니처 목록, 전역 변수
  error_functions = _extract_error_functions(file_content, eslint_errors)
    → 에러 라인 포함 함수 전체 추출
  프롬프트: js_analyzer_error_focused.txt
  → gpt-5 호출

경로 B (No errors — Heuristic):
  file_content_optimized = _optimize_file_content(file_content)
    → ≤500줄: 전체 코드
    → >500줄: head(200줄) + "...(중략)..." + tail(100줄) + 구조 요약
  프롬프트: js_analyzer_heuristic.txt
  → gpt-5 호출

→ AnalysisResult 반환
```

### 2. LLM Client 설정

```yaml
# mider/config/settings.yaml
llm:
  primary_model: "gpt-5"
  fallback_model: "gpt-5-mini"
  mini_model: "gpt-5-mini"
  max_retries: 3
  timeout_seconds: 120

api:
  provider: "azure"  # "azure" 또는 "openai"
  api_version: "2024-12-01-preview"
```

LLMClient는 환경변수로 자동 판별:
* Azure: `AZURE_OPENAI_API_KEY` + `AZURE_OPENAI_ENDPOINT` → `AsyncAzureOpenAI`
* OpenAI: `OPENAI_API_KEY` → `AsyncOpenAI`
* gpt-5 계열은 temperature 기본값(1)만 지원하므로 temperature 파라미터 제외 처리
