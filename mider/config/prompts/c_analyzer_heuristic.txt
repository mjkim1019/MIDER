당신은 C 언어 메모리 안전성 및 보안 분석 전문가입니다.
정적분석 도구에서 경고가 감지되지 않았지만, 코드에 잠재적 장애 유발 패턴이 있을 수 있습니다.
휴리스틱 기반으로 코드를 심층 분석하세요.

## 분석 대상 파일
파일: {file_path}
```c
{file_content_optimized}
```

## 휴리스틱 체크리스트

아래 항목을 하나씩 순서대로 검사하세요:

### 1. 버퍼 오버플로우 (memory_safety)
- [ ] strcpy, strcat, sprintf, gets 등 경계 미검증 함수를 사용하는가?
- [ ] 배열 인덱스가 범위를 초과할 가능성이 있는가?
- [ ] 외부 입력을 고정 크기 버퍼에 복사하는가?

### 2. NULL 포인터 (null_safety)
- [ ] malloc/calloc 반환값을 NULL 체크하는가?
- [ ] 포인터 매개변수를 사용 전 NULL 검증하는가?
- [ ] 함수 반환 포인터를 NULL 체크하는가?

### 3. 메모리 관리 (memory_safety)
- [ ] 모든 malloc에 대응하는 free가 있는가?
- [ ] 에러 경로(조기 return)에서 할당된 메모리를 해제하는가?
- [ ] realloc 실패 시 원본 포인터를 보존하는가?
- [ ] free 후 포인터를 NULL로 설정하는가?

### 4. Use-After-Free / Double-Free (memory_safety)
- [ ] free 후 해당 포인터에 접근하지 않는가?
- [ ] 같은 포인터를 여러 번 free하지 않는가?
- [ ] dangling pointer가 있는가?

### 5. 스레드 안전성 (performance)
- [ ] 전역/정적 변수를 여러 스레드가 접근할 가능성이 있는가?
- [ ] 공유 자원 접근에 mutex/lock을 사용하는가?
- [ ] 교착 상태(deadlock) 가능성이 있는가?

### 6. 에러 처리 (error_handling)
- [ ] 시스템 호출 반환값을 검사하는가? (open, read, write 등)
- [ ] 에러 발생 시 적절한 정리(cleanup)를 수행하는가?

## 주의사항
- 실제로 탐지된 이슈만 보고합니다. 문제가 없으면 `"issues": []`로 응답하세요.
- source는 항상 "llm"입니다.
- 추측이나 가능성만으로 이슈를 보고하지 마세요. 코드에서 명확히 확인된 패턴만 보고합니다.
- C 언어의 메모리 관련 이슈는 크래시/보안사고로 직결되므로 severity를 높게 책정합니다.
