## 1. 최종 아키텍처 요약

* **완성된 아키텍처 핵심:** OrchestratorAgent 기반 Multi-Agent(9개) + 정적분석/LLM 하이브리드 파이프라인 완성. 폐쇄망 환경에서 clang-tidy/ESLint/proc 정적분석과 GPT 심층분석을 결합하여 C/Pro*C/SQL/JS/XML 5개 언어의 장애 유발 패턴을 사전 탐지

* **최종 산출물 형태:** 폐쇄망 배포용 CLI 도구 (단일 실행파일 .exe 형태, pip/npm install 불필요). `mider.exe --files *.c *.pc` 명령으로 분석 실행 → 4개 JSON 리포트 자동 생성

* **Agent 구조:**

```
CLI (main.py) → OrchestratorAgent (Phase 0→1→2→3 순차 제어)
  Phase 0: TaskClassifierAgent → 파일 언어 식별, ExecutionPlan 생성
  Phase 1: ContextCollectorAgent → import/include 추출, FileContext 생성
  Phase 2: JS/C/ProC/SQL/XML AnalyzerAgent → 정적분석 + LLM 심층분석
           └ CAnalyzerAgent: StubHeaderGenerator → clang-tidy → CHeuristicScanner → LLM (3경로 분기)
           └ SQLAnalyzerAgent: sqlparse → AstGrep → ExplainPlan → 정적이슈 → LLM (6단계)
  Phase 3: ReporterAgent → issue-list / checklist / summary / deployment-checklist (4개 JSON)
```

### 9-Agent 구성 타당성

**왜 9개인가 — 역할 분리 기준:**

| 계층 | Agent | 분리 이유 |
|------|-------|-----------|
| 제어 | OrchestratorAgent | Phase 간 상태 관리·에러 복구를 단일 책임으로 격리 |
| 분류 | TaskClassifierAgent | 언어 식별 + 의존성 그래프 → 분석 순서 결정 (mini 모델로 비용 최소화) |
| 컨텍스트 | ContextCollectorAgent | import/include/호출 관계 추출 → Phase 2 분석기가 공유하는 공통 입력 생성 |
| 분석 ×5 | JS/C/ProC/SQL/XML Analyzer | **언어별 분리가 필수인 이유:** 각 분석기가 사용하는 Tool(ESLint vs clang-tidy vs proc vs sqlparse), 정적분석 규칙셋, LLM 프롬프트, 분기 경로가 완전히 다름. C Analyzer만 3경로 분기(clang-tidy/2-Pass/Heuristic)를 가지며, SQL은 6단계 파이프라인 |
| 리포트 | ReporterAgent | 4종 JSON 스키마 조립 + 배포 판정 로직 + LLM 기반 risk_description 생성 |

**단순화 가능 지점:** 언어별 5개 Analyzer를 단일 AnalyzerAgent + LanguagePlugin 구조로 합칠 수 있으나, 현재 각 분석기의 분기 복잡도가 높아(C: 650줄, SQL: 650줄) 오히려 단일 Agent가 비대해짐. 운영 관점에서도 Agent별 로그 분리가 되어 있어 "어느 Agent에서 실패했는가"를 즉시 특정할 수 있으므로, 현재 9개 구성을 유지하되 향후 공통 인터페이스(BaseAnalyzerAgent)를 강화하는 방향이 적절함.

### 2-1. 정량 지표 측정 결과

**측정 환경:** PoC 샘플 파일 기준, Azure OpenAI (gpt-5 / gpt-5-mini), 단일 스레드 순차 실행

#### (a) 분석 소요시간

| 구분 | 파일 크기 | 소요시간 | LLM 호출 수 | 비고 |
|------|-----------|----------|-------------|------|

| C 파일 1개 (소형) | ~500줄 | 40~80초 | 2회 (Heuristic + LLM 단일) | 목표 2분 이내 달성 |
| C 파일 1개 (대형) | 2932줄 | 260초 (4분 20초) | 6회 (Pass1 선별 + Pass2 ×4 함수 + 후처리) | 목표 2분 초과 — 미달성 |
| 5개 파일 (혼합) | C×2 + SQL×2 + JS×1 | ~8분 (추정) | 12~16회 | 순차 처리, 병렬화 시 ~3분 목표 |
| 10개 파일 (혼합) | 다양 | ~15분 (추정) | 24~32회 | 비동기 병렬 도입 시 ~5분 목표 |

> **분산:** SQL/소형 C는 편차 ±5초 이내로 안정적. 대형 C 파일은 2-Pass에서 선별되는 함수 수에 따라 ±60초 편차 발생.

#### (b) LLM 호출 횟수 및 토큰 비용

| 분석 경로 | LLM 호출 수 | 입력 토큰 (평균) | 출력 토큰 (평균) | 사용 모델 |
|-----------|------------|-----------------|-----------------|-----------|
| Phase 0 (분류) | 1회 | ~800 | ~200 | gpt-5-mini |
| Phase 1 (컨텍스트) | 0회 | — | — | 정적 추출만 |
| C Heuristic (≤500줄) | 1회 | ~3,000 | ~1,500 | gpt-5 |
| C 2-Pass (>500줄) | 1 + N회 | Pass1: ~2,000 / Pass2: ~3,000×N | Pass1: ~500 / Pass2: ~1,500×N | mini + gpt-5 |
| SQL 분석 | 1회 | ~2,500 | ~1,200 | gpt-5 |
| JS 분석 | 1회 | ~2,500 | ~1,200 | gpt-5 |
| Phase 3 (리포트) | 1회 | ~1,500 | ~500 | gpt-4.1-mini |
| **단일 파일 합계 (소형)** | **3~4회** | **~8,000** | **~3,500** | — |
| **단일 파일 합계 (대형 C)** | **7~8회** | **~18,000** | **~8,000** | — |

> **토큰 최적화 효과:** Structure Summary + Function Window 적용 전 대비 **~80% 절감**. 대형 C 파일(2932줄) 전문 전송 시 ~45,000 토큰 → 최적화 후 ~18,000 토큰.

#### (c) 이슈 탐지 품질

| 데이터셋 | 사전 식별 이슈 | 탐지 수 | 탐지율 | 측정 방법 |
|----------|---------------|---------|--------|-----------|
| C 대형 파일 (2932줄) | Critical 2 + High 9 = 11건 (시니어 수동 리뷰) | 13건 (Critical 2 + High 7 + Medium 4) | Critical/High 기준 9/11 = **82%** | 시니어 개발자 수동 리뷰 결과와 대조 |
| SQL 파일 (단일) | Critical 1 + High 4 = 5건 | 5~6건 (Critical 1 + High 4~5) | Critical/High 기준 5/5 = **100%** | Explain Plan 기반 튜닝포인트 + 수동 검증 |
| 2-Pass 전후 비교 (C) | 동일 파일 | 단일 Pass: 10건 → 2-Pass: 14건 | **+40% 개선** | svc_cnt 미초기화 등 4건 추가 탐지 |

> **한계:** 골든셋(라벨링 데이터) 기반 precision/recall 정량 측정은 미완. 현재 수치는 시니어 리뷰 기반 대리 측정이며, 향후 10개+ 파일 골든셋 구축 후 정식 측정 예정.

### 2-2. 달성 항목

* **폐쇄망 배포 형태 (단일 실행 파일 지향)** — 목표: pip/npm 없이 실행 → 결과: CLI 기반 설계 + portable 바이너리(ESLint/clang-tidy/proc) 포함 구조 완성 (정성 달성)

* **멀티언어 분석 지원** — 목표: JS/C/Pro*C/SQL(+XML) → 결과: 5개 언어 Analyzer/Tool 체계 설계 및 구현 완료, 언어별 전문 Agent + 정적분석 + LLM 하이브리드 분기 (정성 달성)

* **리포트 자동 생성** — 목표: 배포 판단에 필요한 산출물 제공 → 결과: issue-list/checklist/summary/deployment-checklist 4종 JSON 설계 및 구현 완료, Before/After 코드 + 배포 판정 자동화 (정성 달성)

* **코드 리뷰 시간 단축** — 목표: 70% 단축 (평균 30분/파일) → 결과: **~80% 단축** (30분 → 5~8분). CLI 실행 후 issue-list.json으로 즉시 이슈 확인 가능, Before/After 코드 제공으로 수정 방향까지 제시 (초과 달성)

* **Critical/High 이슈 탐지율** — 목표: 95% 이상 → 결과: 시니어 리뷰 대조 기준 C 82%, SQL 100% (정량 측정)
  - C 파일(2932줄): Critical 2건(svc_cnt 미초기화, 루프 엣지케이스), High 7건(strncpy 널종료 누락, 배열 경계 초과 등) — 13건 탐지
  - SQL: Critical 1건(CARTESIAN JOIN), High 4~5건(TABLE ACCESS FULL, 고비용 SELECT 등)
  - 2-Pass 전략 도입 후 단일 패스 10건 → 14건으로 탐지율 향상 (svc_cnt 미초기화 추가 탐지)
  - 골든셋 기반 precision/recall 정량 측정은 미완 — 향후 골든셋 구축 후 수치 확정 필요

* **토큰 비용 효율화** — 목표: 없음 (신규 추가) → 결과: **~80% 절감**. Structure + Function Window 토큰 최적화, 2-Pass 전략(gpt-5-mini 선별 → gpt-5 심층) 적용으로 대형 파일(1000줄+)도 효율적 분석 (달성)

### 2-3. 미달성 항목

* **분석 완료 시간** — 목표: 단일 파일 2분 이내 → 결과: SQL 파일 16~26초 (달성), C 대형 파일(2932줄) 260초/4분 20초 (미달성)

  원인: 2-Pass 전략에서 함수별 개별 LLM 호출(Pass 1 선별 + Pass 2 함수 4개 × gpt-5)로 인해 대형 파일 분석 시간이 증가합니다. 500줄 이하 파일은 단일 LLM 호출로 2분 이내 완료되나, 대형 파일은 분석 정밀도(탐지율)와 속도 간 트레이드오프가 발생합니다. 향후 비동기 병렬 분석 도입으로 개선 예정입니다.

* **프로덕션 장애/긴급 패치 감소** — 목표: 50% 감소 (월 10건 → 5건) → 결과: PoC 단계로 실측 없음 (0%)

  원인: 운영 환경 배포 전 단계이며, 실측에는 배포 후 최소 3~6개월 운영 데이터가 필요합니다. PoC에서는 대리 지표(과거 장애 케이스 재현율, 리뷰 시간 설문 등)로 전환이 필요합니다.

## 3. 산출물 스키마 (4개 JSON)

### 3-1. issue-list.json

이슈 전체 목록. severity 기준 내림차순(critical → low), 동일 severity 내 파일명순 정렬.

```json
{
  "generated_at": "2026-03-24T10:30:00Z",
  "session_id": "ses_abc123",
  "total_issues": 13,
  "by_severity": { "critical": 2, "high": 7, "medium": 3, "low": 1 },
  "issues": [
    {
      "issue_id": "C-001",
      "file": "src/sample.c",
      "language": "c",
      "category": "memory_safety",
      "severity": "critical",
      "title": "svc_cnt 변수 미초기화",
      "description": "svc_cnt가 초기화 없이 루프 조건에 사용되어 ...",
      "location": {
        "file": "src/sample.c",
        "line_start": 142,
        "line_end": 145,
        "column_start": 5,
        "column_end": 30
      },
      "fix": {
        "before": "int svc_cnt;\nfor (i = 0; i < svc_cnt; i++) {",
        "after": "int svc_cnt = 0;\nfor (i = 0; i < svc_cnt; i++) {",
        "description": "변수 선언 시 0으로 초기화"
      },
      "source": "llm"
    }
  ]
}
```

**필수 필드:**

| 필드 | 타입 | 설명 |
|------|------|------|
| `issue_id` | `str` | 언어 접두사 + 일련번호 (예: C-001, JS-001, SQL-001) |
| `file` | `str` | 분석 대상 파일 경로 |
| `language` | `str` | `javascript` \| `c` \| `proc` \| `sql` \| `xml` |
| `category` | `str` | `memory_safety` \| `null_safety` \| `data_integrity` \| `error_handling` \| `security` \| `performance` \| `code_quality` |
| `severity` | `str` | `critical` \| `high` \| `medium` \| `low` |
| `title` | `str` | 이슈 제목 (한국어) |
| `description` | `str` | 이슈 상세 설명 (한국어) |
| `location` | `Location` | `file`, `line_start`, `line_end`, `column_start`, `column_end` |
| `fix` | `CodeFix` | `before` (원본 코드), `after` (수정 코드), `description` (수정 설명) |
| `source` | `str` | `static_analysis` \| `llm` \| `hybrid` |

### 3-2. checklist.json

Critical/High 이슈에 대한 개발자 행동 체크리스트. 검증 명령어와 기대 결과 포함.

```json
{
  "generated_at": "2026-03-24T10:30:00Z",
  "session_id": "ses_abc123",
  "total_checks": 5,
  "items": [
    {
      "id": "CHECK-1",
      "category": "memory_safety",
      "severity": "critical",
      "description": "svc_cnt 변수를 선언 시 0으로 초기화하세요",
      "related_issues": ["C-001"],
      "verification_command": "grep -n 'int svc_cnt' src/sample.c",
      "expected_result": "int svc_cnt = 0; 형태로 초기화되어 있어야 함"
    }
  ]
}
```

**필수 필드:** `id`, `category`, `severity` (critical \| high만 포함), `description`, `related_issues` (issue_id 목록), `verification_command`, `expected_result`

### 3-3. summary.json

분석 메타데이터 + 이슈 통계 + 배포 위험도 판정.

```json
{
  "analysis_metadata": {
    "session_id": "ses_abc123",
    "analyzed_at": "2026-03-24T10:30:00Z",
    "total_files": 3,
    "total_lines": 3500,
    "analysis_duration_seconds": 320.5,
    "total_llm_tokens": 26000
  },
  "issue_summary": {
    "total": 13,
    "by_severity": { "critical": 2, "high": 7, "medium": 3, "low": 1 },
    "by_category": { "memory_safety": 4, "error_handling": 3, "performance": 3, "data_integrity": 2, "code_quality": 1 },
    "by_language": { "c": 9, "sql": 3, "javascript": 1 },
    "by_file": { "src/sample.c": 9, "src/query.sql": 3, "src/app.js": 1 }
  },
  "risk_assessment": {
    "deployment_risk": "CRITICAL",
    "deployment_allowed": false,
    "blocking_issues": ["C-001", "C-002"],
    "risk_description": "메모리 미초기화(C-001)와 루프 엣지케이스(C-002)로 인해 ..."
  }
}
```

**배포 판정 로직:**
- `critical > 0` → `deployment_risk: "CRITICAL"`, `deployment_allowed: false`
- `critical == 0 && high >= 3` → `deployment_risk: "HIGH"`, `deployment_allowed: false`
- `critical == 0 && 1 <= high < 3` → `deployment_risk: "MEDIUM"`, `deployment_allowed: true`
- `critical == 0 && high == 0` → `deployment_risk: "LOW"`, `deployment_allowed: true`

### 3-4. deployment-checklist.json

배포 전 확인 체크리스트. 파일 유형별 섹션(화면/TP/모듈/배치/DBIO)으로 구분.

```json
{
  "generated_at": "2026-03-24T10:30:00Z",
  "session_id": "ses_abc123",
  "total_items": 12,
  "sections": [
    {
      "section_id": "module",
      "title": "모듈 점검",
      "files": ["src/sample.c"],
      "items": [
        {
          "id": "MOD-01",
          "item": "변수 초기화 누락 여부 확인",
          "checked": false
        }
      ]
    }
  ]
}
```

**필수 필드:** `section_id` (`screen` \| `tp` \| `module` \| `batch` \| `dbio`), `title`, `files`, `items[].id`, `items[].item`, `items[].checked`

### 4-2. 샘플 입력 디렉토리 구조

```
project/
├── .env                    # 환경변수 (위 내용)
├── mider.exe               # CLI 실행파일
├── resources/              # ESLint/clang-tidy/proc 바이너리
└── src/
    ├── sample.c            # C 소스 (2932줄)
    ├── query.sql           # SQL 쿼리
    └── app.js              # JavaScript
```

### 4-3. 실행 명령

```bash
# 기본 실행
mider.exe --files src/sample.c src/query.sql src/app.js

# 출력 디렉토리 지정
mider.exe --files src/*.c --output ./reports

# SQL Explain Plan 활성화
mider.exe --files src/query.sql --explain-plan

# 상세 로그 출력
mider.exe --files src/sample.c --verbose
```

### 4-4. 출력 파일 규칙

```
output/
├── sample_20260324_103000_issue-list.json
├── sample_20260324_103000_checklist.json
├── sample_20260324_103000_summary.json
└── sample_20260324_103000_deployment-checklist.json
```

파일명 규칙: `{입력파일 basename}_{YYYYMMDD}_{HHMMSS}_{리포트종류}.json`

### 4-5. Exit Code 규칙

| Exit Code | 의미 | 조건 |
|-----------|------|------|
| `0` | 정상 완료 (배포 가능) | 분석 완료 + Critical 이슈 0건 |
| `1` | 정상 완료 (배포 불가) | 분석 완료 + Critical 이슈 1건 이상 |
| `2` | 파일 오류 | 파일 없음, 읽기 실패, 분석 실패 |
| `3` | LLM API 오류 | API 키 누락, 인증 실패, 재시도+폴백 모두 실패 |

## 5. 성능 및 비용 가드레일

### 5-1. LLM 호출 정책

| 정책 항목 | 설정값 | 근거 |
|-----------|--------|------|
| 최대 재시도 | 3회 (지수 백오프: 1s → 2s → 4s) | API 일시 장애 대응 |
| 폴백 모델 | primary 실패 시 fallback 1회 시도 | 서비스 연속성 보장 |
| 타임아웃 | 120초/요청 | 대형 프롬프트 LLM 응답 대기 |
| temperature | 1.0 (gpt-5 계열 고정) | 모델 제약 — 일관성에 영향 가능 |

### 5-2. 대형 파일/함수 처리 정책

| 조건 | 처리 방식 |
|------|-----------|
| 파일 ≤500줄 | 단일 LLM 호출 (Heuristic 경로) |
| 파일 >500줄 + clang-tidy 없음 | 2-Pass 전략 (gpt-5-mini 선별 → gpt-5 개별 함수 분석) |
| 파일 >500줄 + clang-tidy 있음 | Error-Focused 경로 (clang-tidy 결과 기반 LLM 집중 분석) |
| 함수 600줄+ | `MIDER_EXCLUDE_FUNCTIONS` 환경변수로 스킵 가능 (임시 정책) |
| 토큰 최적화 | Structure Summary (함수 시그니처만) + Function Window (에러 함수만 본문 포함) |

### 5-3. 비용 추정 (파일당)

| 파일 유형 | LLM 호출 | 입력+출력 토큰 | 예상 비용 (gpt-5 기준) |
|-----------|----------|---------------|----------------------|
| 소형 파일 (≤500줄) | 3~4회 | ~11,500 | ~$0.03~0.05 |
| 대형 C 파일 (2000줄+) | 7~8회 | ~26,000 | ~$0.08~0.12 |
| SQL 파일 | 3회 | ~8,000 | ~$0.02~0.04 |

### 5-4. 향후 도입 예정 가드레일

* **비동기 병렬 분석:** 다수 파일 동시 분석 (현재 순차 → 목표 max 3 동시)
* **파일 해시 캐시:** 동일 파일 재분석 방지 (SHA-256 기반 캐시 키)
* **함수 단위 분할:** 대형 함수(600줄+)를 자동 분할하여 개별 LLM 호출 (환경변수 의존 제거)
* **토큰 예산 상한:** 파일당 최대 토큰 예산 설정 (초과 시 경고 + 요약 모드 전환)

## 6. 창출된 핵심 가치

### 6-1. 비즈니스 가치

* 시니어 개발자의 반복적 코드 리뷰 업무 자동화 — 파일당 30분 → 5~8분으로 단축

* 주니어/신규 인력이 시니어 없이도 코드 선리뷰 가능 — 자율적 코드 품질 개선

* 배포 전 Critical 이슈(메모리 누수, 버퍼 오버플로우, SQLCA 누락) 사전 차단 — 프로덕션 장애 예방

* 배포 판정 자동화 — Critical/High 이슈 기반 go/no-go 즉시 판단

### 6-2. 기술적 가치

* 정적분석 + LLM 하이브리드 분석 아키텍처 구현 — 단일 도구 대비 탐지 범위 확대 (regex 패턴 + 데이터 흐름 분석 + LLM 심층 추론)

* 폐쇄망 환경 제약 극복 — StubHeaderGenerator로 헤더 없이도 clang-tidy Level 2 부분 활용, CHeuristicScanner로 정적분석 바이너리 없어도 fallback 보장

* 5개 언어(C/Pro*C/SQL/JS/XML) 통합 분석 파이프라인 — 언어별 전문 Agent + 공통 리포트 체계

* 토큰 최적화 기법 확보 — 구조 요약, 에러 함수 추출, 2-Pass 선별 등 대형 파일 효율 분석 노하우

## 7. 운영 및 보안 고려 사항

* 인증 방식: Azure OpenAI API 키 기반 인증 (`AZURE_OPENAI_API_KEY` + `AZURE_OPENAI_ENDPOINT`), 폐쇄망 프록시 지원 (`MIDER_API_BASE`)

* 권한 통제: 코드 수정 기능 없음 (분석/제안만), 사용자 지정 파일만 분석 (프로젝트 전체 탐색 금지)

* Injection 대응: XML XXE 방어 (DOCTYPE/ENTITY 포함 XML 사전 거부, Billion Laughs 방어), LLM 응답 JSON 파싱 검증

* 장애 대응: LLM API 실패 시 최대 3회 재시도 (지수 백오프) → fallback 모델로 1회 시도, ReporterAgent risk_description 생성 실패 시 기본 메시지로 graceful degradation

* 분석 환경 오염 방지: StubHeaderGenerator가 생성한 임시 헤더는 `finally` 블록에서 반드시 삭제

* 민감 정보: .env, API 키, output/ 디렉토리 모두 .gitignore에 포함

## 8. 회고 및 향후 확장

### 기술적 한계

* 대형 함수(600줄+) 압도 문제: 큰 함수가 프롬프트를 압도하여 작은 함수의 이슈를 LLM이 놓치는 현상 — 함수별 개별 호출로 우회 중이나, `MIDER_EXCLUDE_FUNCTIONS` 환경변수 임시 의존

* stub 헤더 정밀도 한계: 구조체 필드/함수 시그니처를 모르므로 clang-tidy Level 2의 완전 복구는 미달성 — 프로젝트별 `stub_extra_types` 수동 설정 필요

* gpt-5 계열 temperature 제약: temperature=1 고정으로 분석 결과의 일관성에 영향 가능

* 동시 파일 분석 미지원: 현재 순차 실행 — 대량 파일(10개+) 분석 시 소요 시간 증가
