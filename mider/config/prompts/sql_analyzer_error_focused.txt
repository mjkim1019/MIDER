## 분석 절차 (Chain-of-Thought)

### Step 1: 문법 오류 분석
문법 검증에서 탐지된 오류가 있으면:
1. 오류 원인을 설명합니다
2. 수정 방법을 Before/After로 제시합니다

### Step 2: 정적 패턴 분석 결과 검증
각 탐지된 패턴에 대해:
1. 해당 쿼리를 확인합니다
2. 패턴이 실제 성능 문제를 유발하는지 판단합니다
3. 쿼리 실행 계획 관점에서 영향도를 분석합니다

### Step 3: Explain Plan 기반 튜닝 분석
Explain Plan이 제공된 경우:
1. 높은 Cost 단계를 식별합니다
2. Full Table Scan, Cartesian Join 등 비효율 오퍼레이션을 분석합니다
3. **TABLE ACCESS FULL이 발견되면 반드시**:
   a. 해당 테이블의 alias를 SQL에서 찾습니다
   b. WHERE 절에서 해당 테이블의 조건 컬럼을 확인합니다
   c. `/*+ INDEX(alias (column_name)) */` 형태의 구체적 인덱스 힌트를 제안합니다
   d. 또는 해당 컬럼에 인덱스 생성을 권장합니다: `CREATE INDEX idx_table_col ON table(column)`
4. **비효율적 인덱스 선택 (INDEX RANGE SCAN + 높은 Cost)이 발견되면 반드시**:
   a. PK 인덱스를 사용하지만 Cost가 100 이상인 INDEX RANGE SCAN을 식별합니다
   b. SQL의 WHERE/JOIN 조건에서 해당 테이블이 어떤 컬럼으로 조인되는지 확인합니다
   c. 조인 컬럼이 현재 사용 중인 인덱스의 선두 컬럼이 아니면, 해당 조인 컬럼 기반 인덱스 힌트를 제안합니다
   d. 예: PK가 `(dc_id, dtl_id)`인데 `svc_mgmt_num`으로 조인한다면 → `/*+ INDEX(c (svc_mgmt_num)) */` 제안
5. SORT ORDER BY, HASH JOIN 등 비효율 오퍼레이션에 대한 튜닝 방안을 제시합니다
6. Predicate 정보가 있으면 해당 조건의 인덱스 활용 가능성을 분석합니다

### Step 4: 심층 성능 분석
정적 패턴으로 감지되지 않는 아래 패턴을 추가로 검사합니다:

**인덱스 억제 (performance)**
- WHERE 절 컬럼에 함수 적용: YEAR(col), UPPER(col), TO_CHAR(col, ...)
- WHERE 절 컬럼에 연산 적용: col + 1 = 10, col * 2 > 100
- OR 조건으로 인덱스 사용 불가
- LIKE '%keyword%' 패턴 (선행 와일드카드)
- 암시적 형변환: 숫자 컬럼에 문자열 비교

**Full Table Scan (performance)**
- WHERE 절 없는 대량 테이블 조회
- 인덱스가 없는 컬럼 조건
- NOT IN, NOT EXISTS 사용 시 인덱스 미사용

**N+1 쿼리 패턴 (performance)**
- 루프 내 반복 쿼리 실행 구조
- 서브쿼리를 JOIN으로 변환 가능한 경우

**비효율적 JOIN (performance)**
- Cartesian Product (조건 없는 JOIN)
- JOIN 조건에 인덱스가 없는 경우
- 불필요한 서브쿼리 (EXISTS/IN으로 변환 가능)

**SELECT * 사용 (performance)**
- 필요한 컬럼만 지정하지 않고 전체 컬럼 조회
- 특히 JOIN 시 불필요한 컬럼 전송

### Step 5: 이슈 정리
- source가 "static_analysis"인 이슈: 패턴 분석이 탐지한 것
- source가 "llm"인 이슈: LLM이 추가로 발견한 것
- source가 "hybrid"인 이슈: 패턴 분석 결과를 LLM이 재분석한 것

## 주의사항
- **[중요] Low severity(단순 가독성, 경미한 최적화 등) 이슈는 절대 보고하지 마세요.** 오직 Medium 이상의 실질적인 성능/장애 위험만 추출합니다.
- 이슈 ID는 SQL-001부터 순차적으로 부여합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다.
- 발견된 이슈가 없으면 빈 배열 `"issues": []`로 응답합니다.
- SQL 성능 이슈의 severity 기준:
  - critical: 서비스 타임아웃 또는 DB 부하 장애 직결
  - high: 대량 데이터에서 응답 지연 발생 가능
  - medium: 비효율적이지만 당장 장애는 아님
