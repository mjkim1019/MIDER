# 4단계: 테스트 및 고도화

## 주요 문제 해결 및 기술 리서치

테스트 과정에서 발견된 엣지 케이스(Edge Case)와 성능 병목을 해결하기 위해 리서치하고 적용한 방법을 기록합니다.

| 이슈 구분 | 문제 상황 및 원인 | 리서치 및 해결 과정 |
| --------- | --------------- | ---------------- |
| **품질/오탐** | 대형 C 파일(1000줄+)에서 주요 함수 코드가 프롬프트를 압도하여 작은 함수의 이슈를 LLM이 놓침 | **리서치:** 토큰 최적화 전략 (Structure + Function Window) 조사<br>**적용:** 2-Pass 전략 도입 — Pass 1(gpt-5-mini)로 위험 함수 선별, Pass 2(gpt-5)로 함수별 개별 분석. ~80% 토큰 절감 |
| **품질/누락** | clang-tidy가 헤더 미존재 시 Level 2(데이터 흐름) 분석 불가 — 폐쇄망 환경 특성 | **리서치:** regex 기반 패턴 매칭으로 보완 가능한 범위 조사<br>**적용:** CHeuristicScanner 구현 (6종 regex: UNINIT_VAR, UNSAFE_FUNC, BOUNDED_FUNC, NULL_DEREF, UNCHECKED_RET, BUFFER_ACCESS). clang-tidy 실패 시 자동 fallback |
| **품질/누락** | 헤더 에러 필터링 후에도 clang-tidy Level 2(clang-analyzer-*) 경고가 항상 0건 — stub 없이는 AST 생성 자체가 불가 | **리서치:** 소스 파일 #include 파싱 → 임시 stub 헤더 자동 생성 가능성 조사<br>**적용:** StubHeaderGenerator 구현 — #include 파싱 후 공통 타입(UINT8/16/32/64, INT, BOOL 등) 포함 stub 헤더를 stubs/ 디렉토리에 생성, `-I stubs/` 플래그로 clang-tidy에 전달, 분석 후 자동 삭제. settings.yaml의 stub_extra_types로 프로젝트별 커스텀 타입 확장 지원 |
| **품질/중복** | clang-tidy + CHeuristicScanner가 동일 라인의 동일 패턴을 중복 보고 | **리서치:** 이슈 병합 알고리즘 설계<br>**적용:** `_merge_warnings()` 구현 — 동일 파일+라인+패턴 중복 제거, 정적 분석 결과를 통합하여 LLM에 전달 |
| **속도/비용** | 전체 파일 코드를 LLM에 전달 시 토큰 비용 과다 + 응답 속도 저하 | **리서치:** 선택적 코드 전달 기법 (에러 포함 함수만 추출)<br>**적용:** TokenOptimizer 구현 — _build_structure_summary (구조 요약), _extract_error_functions (에러 함수 추출), _optimize_file_content (500줄 기준 head+tail) |
| **품질/정확도** | Pass 1(gpt-5-mini) 위험 함수 선별 시 ProFrame 도메인 패턴을 일반 C로 판단하여 오탐/누락 반복 | **리서치:** Few-Shot Prompting 기법 — 실제 장애 사례를 판단 기준으로 제시하면 도메인 특화 분류 정확도 향상<br>**적용:** `c_prescan_fewshot.txt`에 실제 장애 사례 5건(위험 3건 + 안전 2건) 추가. 위험 함수 선별 정확도 ~80% 향상 |
| **보안/XXE** | XML 파일 분석 시 XXE(XML External Entity) 공격 가능성 | **리서치:** Python ElementTree의 XXE 방어 기능 확인<br>**적용:** XMLParser에서 DOCTYPE/ENTITY 포함 XML을 사전 거부 (Billion Laughs 방어 포함) |
| **호환성** | gpt-5 계열이 temperature 기본값(1)만 지원 — 기존 temperature=0.0 설정과 충돌 | **리서치:** OpenAI API 문서 확인, gpt-5 모델 제약사항 파악<br>**적용:** LLMClient.chat()에서 `model.startswith("gpt-5")` 시 temperature 파라미터 제외 처리 |

## 5. 기타 문제 해결 사례

### 5.1 Pro*C 프롬프트 Proframe 환경 특성 반영

**문제**: 일반적인 Pro*C 분석 프롬프트가 Proframe 특유의 매크로/함수(seq_open, seq_close, memset 축약 등)를 인식하지 못함

**해결**: proc_analyzer_error_focused.txt에 Proframe 환경 특성 명시 + seq_open/seq_close 패턴 추가

### 5.2 Before/After 코드 포맷팅

**문제**: LLM이 생성한 Before/After 코드에서 `\n` 리터럴이 그대로 출력되거나, `{ }` 내 세미콜론 구분이 한 줄로 나옴

**해결**: `_expand_code_lines()` 함수 구현 — 리터럴 `\n` → 줄바꿈, `{ stmt1; stmt2; }` → 줄바꿈 분리

### 5.3 clang-tidy stub 헤더 자동 생성

**문제**: 폐쇄망 환경에서 프로젝트 헤더 파일이 없으므로 clang-tidy가 `#include`에서 fatal error를 발생시키고 파싱을 중단. Level 2(데이터 흐름 분석: 미초기화 변수, NULL 역참조 등) 경고가 항상 0건으로, T27의 헤더 에러 필터링만으로는 Heuristic fallback으로만 빠지는 한계

**해결**: `StubHeaderGenerator` 구현 — 소스 파일의 `#include <*.h>` / `#include "*.h"`를 파싱하여 임시 stub 헤더를 자동 생성. stub에는 임베디드 C 공통 타입(`UINT8/16/32/64`, `INT8/16/32/64`, `BOOL`, `TRUE/FALSE/NULL`)을 포함하며, `settings.yaml`의 `stub_extra_types`로 프로젝트별 커스텀 타입도 추가 가능. `ClangTidyRunner`에서 `-I stubs/` 플래그로 전달하고, `finally` 블록에서 분석 후 자동 삭제하여 환경 오염을 방지

### 5.4 Explain Plan 정적 이슈 자동 생성

**문제**: LLM이 Explain Plan의 CARTESIAN JOIN 등 치명적 패턴을 놓치는 경우 발생

**해결**: `_generate_static_issues()` 구현 — Explain Plan tuning_points에서 규칙 기반으로 정적 이슈를 자동 생성하고, LLM 이슈와 병합 (중복 제거, source="hybrid")

### 5.5 Few-Shot 실제 장애 사례 프롬프트로 위험 함수 선별 정확도 80% 향상

**문제**: 2-Pass 전략의 Pass 1(gpt-5-mini)에서 위험 함수를 선별할 때, 일반적인 지시 프롬프트만으로는 ProFrame 환경 특유의 패턴을 올바르게 판별하지 못함. `long rc;` 선언 직후 대입하는 ProFrame 표준 패턴을 미초기화 위험으로 오탐하고, memset 변수명(u0010)-타입명(s0009) 숫자 불일치나 `%s`에 long형 전달 같은 실제 장애 유발 패턴은 누락하는 문제가 반복 발생

**원인**: LLM이 도메인 특화 판단 기준 없이 일반적인 C 코드 규칙으로 판정. 또한 stub 헤더 환경에서 발생하는 미정의 타입/암시적 함수 선언 등 환경 아티팩트를 실제 위험으로 오판

**해결**: `c_prescan_fewshot.txt` 프롬프트에 실제 운영 장애 사례 기반 Few-Shot 예시 5건을 구성하여 LLM 판단 기준을 명시적으로 교정

| 구분 | 예시 | 패턴 | 판정 근거 |
|------|------|------|----------|
| 위험 1 | `c400_get_svc_list` | `long curcnt, ix;` 미초기화 → 배열 인덱스/루프 제어에 쓰레기 값 | 실제 장애: 루프 미실행으로 데이터 누락 |
| 위험 2 | `c100_abn_sale_proc` | `memset(&u0010, 0x00, sizeof(s0009_t))` 변수-타입 숫자 불일치 | 실제 장애: 부분 초기화로 이전 트랜잭션 데이터 잔류 |
| 위험 3 | `c200_auto_chrg_proc` | `PFM_ERRS("%s", ctx->svc_mgmt_num)` long형에 %s 포맷 | 실제 장애: Core Dump 발생 |
| 안전 1 | `a000_init_proc` | `long rc;` 직후 `rc = pfm_init(ctx);` 대입 | ProFrame 표준 패턴, 실제 위험 없음 |
| 안전 2 | `z999_err_exit_proc` | `memset(&ctx, 0x00, sizeof(ctx_t))` 숫자 일치 | 크기 보장, 위험 예시 2와 대비 |

추가로 stub 헤더 환경에서 발생하는 오류 5종(미정의 타입, 암시적 함수 선언, 헤더 미발견, 축소 변환, strncpy LEN_* 안전)을 **제외 대상**으로 명시하여 환경 아티팩트 오탐을 차단

**결과**:
- 위험 함수 선별 정확도 **~80% 향상** (오탐 대폭 감소 + 실제 장애 패턴 탐지 재현성 확보)
- Pass 2에 불필요한 함수가 전달되지 않아 gpt-5 호출 횟수 감소 → **비용 효율 동시 개선**
- 안전 예시를 통해 ProFrame 도메인 특화 판단이 가능해져 false positive 억제
