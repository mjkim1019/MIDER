XML 파서 결과를 기반으로 WebSquare 화면 정의의 구조 이슈를 찾으세요.
인라인 JS 코드 분석은 별도 수행되므로, XML 구조에 집중하세요.

## 전처리 참고사항
코드 내 숫자 사이에 `*`가 삽입된 패턴(예: `1030*2300`, `010*1234*5678`)은 개인정보(PII) 검출 필터 우회를 위한 전처리 치환 결과입니다. 이를 타입 오류, 연산자 오용, 또는 코드 결함으로 판단하지 마세요.

## 분석 절차 (Chain-of-Thought)

### Step 1: 파서 오류 분석
1. 중복 ID: 같은 ID가 여러 컴포넌트에 사용됨 → `$w.getById()` 오작동
2. JS 핸들러 누락: ev:onclick 등으로 바인딩했으나 JS에 함수 미정의
3. XML 파싱 오류: 잘못된 태그 구조, 네임스페이스 오류

### Step 2: 심층 구조 분석

**데이터 바인딩 (data_integrity)**
- dataList에 정의된 컬럼이 실제 화면 컴포넌트와 바인딩되지 않음
- dataType이 화면 입력 유형과 불일치 (예: number 타입에 text 입력)
- **화면 데이터 출력 전 초기화 누락**: 조회 결과를 그리드나 리스트에 바인딩하기 전, 기존 데이터를 삭제하는 `removeAll()`, `clear()`, `reset()` 등의 로직이 주석 처리되었거나 누락되었는지 확인 (이전 데이터 잔존 위험)

**이벤트 처리 (error_handling)**
- 복수 이벤트가 동일 함수를 호출하여 의도치 않은 중복 실행
- 이벤트 핸들러 함수명 네이밍 규칙 위반
- 이벤트 핸들러에서 에러 처리 누락

**보안 (security)**
- 사용자 입력 컴포넌트에 유효성 검사(validation) 설정 누락
- 숨김(hidden) 필드에 민감 정보 직접 바인딩

**성능 (performance)**
- 불필요하게 많은 dataList 정의 (실제 사용하지 않는 데이터 리스트)
- 과도한 이벤트 바인딩 (같은 요소에 중복 이벤트)

## 이슈 예시
중복 ID 데이터에 "lines" 필드가 있으면 반드시 location.line_start에 첫 번째 라인, line_end에 마지막 라인을 사용하세요.
{{
  "issue_id": "XML-001",
  "category": "data_integrity",
  "severity": "high",
  "title": "중복 컴포넌트 ID로 인한 런타임 오작동 위험",
  "description": "ID 'btn_search'가 라인 120, 250에서 2개 컴포넌트에 사용되고 있습니다.",
  "location": {{
    "file": "{file_path}",
    "line_start": 120,
    "line_end": 250
  }},
  "fix": {{
    "before": "<trigger id=\"btn_search\" ev:onclick=\"scwin.btn_search_onclick\"/>",
    "after": "<trigger id=\"btn_search_sub\" ev:onclick=\"scwin.btn_search_sub_onclick\"/>",
    "description": "두 번째 컴포넌트의 ID를 고유하게 변경하고 핸들러도 분리합니다."
  }},
  "source": "static_analysis"
}}

## 주의사항
- **[중요] Low severity(단순 가독성, 스타일 등) 이슈는 절대 보고하지 마세요.** 오직 Medium 이상의 실질적인 구조/데이터 결함만 추출합니다.
- 이슈 ID는 XML-001부터 순차적으로 부여합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다.
- 발견된 이슈가 없으면 빈 배열 `"issues": []`로 응답합니다.
- 중복 ID, 핸들러 누락은 source: "static_analysis"로 표시합니다.
- **[중요] 서로 다른 dataList 간 동명 컬럼은 절대 이슈로 보고하지 마세요.** 각 dataList는 완전히 독립된 스코프입니다.
- severity 기준:
  - critical: 화면 렌더링 실패 또는 데이터 손실
  - high: 특정 조건에서 기능 오작동
  - medium: 구조적 품질 저하, 잠재적 버그
