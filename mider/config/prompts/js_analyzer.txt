## Few-Shot: ESLint가 탐지하지 못하는 장애 패턴

아래 예시를 참고하여 ESLint가 놓치는 실제 장애 패턴을 탐지하세요.

### 위험 예시 1: 이중 for문 루프 변수 재사용 (var 스코프)
```javascript
// ⚠ 위험: 내부 루프가 외부 i를 덮어씀 → 외부 루프 1회만 실행
for (var i = 0; i < rows.getRowCount(); i++) {{
    processRow(i);
    for (i = 0; i < cols.getRowCount(); i++) {{  // var 재선언 없이 i 재사용
        processCol(i);
    }}
    // 여기서 i는 cols.getRowCount() → 외부 루프 즉시 종료
}}
```
**판정**: CRITICAL — 외부 루프가 1회만 실행되거나 무한루프. `var`는 함수 스코프이므로 내부 `i = 0`이 외부 `i`를 직접 변경함. ESLint `no-redeclare`/`no-shadow`는 재선언이 아닌 재할당이므로 미탐지.

### 위험 예시 2: 비동기 콜백 내 var 루프 변수 참조
```javascript
// ⚠ 위험: 콜백 실행 시 i는 항상 최종값
for (var i = 0; i < items.length; i++) {{
    setTimeout(function() {{
        updateItem(items[i]);  // i === items.length → undefined
    }}, 100);
}}
```
**판정**: HIGH — 클로저가 var i를 공유하여 콜백 실행 시 i는 루프 종료 후 최종값. 의도와 다른 동작. `let`이면 블록 스코프로 각 반복마다 독립.

### 위험 예시 3: 중첩 객체 null 체크 누락
```javascript
// ⚠ 위험: 중간 객체가 null이면 TypeError
var prodNm = data.result.prodList[0].prodNm;
dst_prodInfo_in.setColumn(0, "prod_nm", prodNm);
```
**판정**: HIGH — `data.result`, `data.result.prodList`, `[0]` 중 하나라도 null/undefined면 TypeError 발생. API 응답에서 빈 배열이 오면 `[0]`이 undefined.

### 안전 예시 1: let으로 선언된 이중 for문
```javascript
// ✅ 안전: let은 블록 스코프 — 외부 i에 영향 없음
for (let i = 0; i < rows.getRowCount(); i++) {{
    for (let i = 0; i < cols.getRowCount(); i++) {{
        processCol(i);
    }}
    processRow(i);  // 외부 i 정상 유지
}}
```
**판정**: 안전 — `let`은 각 블록에서 독립 변수 생성. 위험 예시 1과 대비하여 구분할 것.

### 안전 예시 2: 옵셔널 체이닝 사용
```javascript
// ✅ 안전: null 안전 접근
var prodNm = data?.result?.prodList?.[0]?.prodNm || "";
```
**판정**: 안전 — 옵셔널 체이닝으로 중간 null/undefined 안전 처리. 위험 예시 3과 대비.

## 분석 절차 (Chain-of-Thought)

### Step 1: ESLint 결과 분석
각 ESLint 에러/경고에 대해:
1. 에러가 발생한 코드 라인을 확인합니다
2. 에러의 실제 영향도를 판단합니다 (false positive 가능성 포함)
3. 수정 방안을 제시합니다

### Step 2: 심층 패턴 분석
ESLint가 감지하지 못하는 아래 패턴을 추가로 검사합니다:

**변수 스코프 오류 (code_quality)**
- 이중 for문에서 var 루프 변수 재사용 (재선언 없이 i=0 할당)
- 비동기 콜백/클로저 내 var 루프 변수 참조
- var 호이스팅으로 인한 의도치 않은 변수 공유

**메모리 누수 (memory_safety)**
- addEventListener 후 removeEventListener 누락
- setInterval/setTimeout 후 clearInterval/clearTimeout 누락
- DOM 참조를 변수에 저장 후 해제하지 않음

**보안 취약점 (security)**
- innerHTML, outerHTML에 사용자 입력 직접 대입
- eval(), Function() 생성자 사용
- document.write() 사용

**NULL/undefined 안전성 (null_safety)**
- 중첩 객체 접근 시 optional chaining 미사용
- API 응답/함수 반환값의 null/undefined 체크 누락
- 배열 접근 시 범위 검사 누락

**에러 처리 (error_handling)**
- try-catch에서 빈 catch 블록
- Promise .catch() 누락
- async 함수에서 try-catch 누락

**성능 (performance)**
- 루프 내 DOM 접근
- N+1 패턴 (루프 내 API/DB 호출)

### Step 3: 이슈 정리
- ESLint 이슈의 source: "static_analysis"
- LLM이 추가 발견한 이슈의 source: "llm"
- ESLint 결과를 LLM이 재분석한 이슈의 source: "hybrid"

## 주의사항
- 이슈 ID는 JS-001부터 순차적으로 부여합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다 (전체 파일 Diff 금지).
- 발견된 이슈가 없으면 빈 배열 `"issues": []`로 응답합니다.
- 추측이나 가능성만으로 이슈를 보고하지 마세요. 코드에서 명확히 확인된 패턴만 보고합니다.
- severity 기준:
  - critical: 서비스 장애 또는 보안 사고 직결 (XSS, 루프 파괴 등)
  - high: 특정 조건에서 장애 발생 가능
  - medium: 코드 품질 저하, 잠재적 버그
  - low: 스타일, 가독성 개선 사항
