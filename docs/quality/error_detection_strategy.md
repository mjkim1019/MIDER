# 오류 탐지 레벨 필터링 전략 (이중 필터링)

본 문서는 MIDER 시스템에서 `Low` 등급 이슈를 노출하지 않고 핵심 이슈에 집중하기 위해 도입된 **이중 필터링(Double Filtering)** 전략에 대해 설명합니다.

## 1. 도입 배경
분석 과정에서 LLM이 생성하는 `Low` 등급 이슈(단순 가독성, 주석 누락, 스타일 가이드 등)는 실제 장애 예방보다는 노이즈로 작용하는 경우가 많았습니다. 이를 해결하기 위해 다음과 같은 목표를 설정했습니다.
- **가독성 향상**: 리포트에서 실제 대응이 필요한 `Medium` 이상의 이슈만 노출.
- **일관성 확보**: Phase 2(분석 로그)와 Phase 3(최종 리포트)의 결과 카운트를 100% 일치시킴.
- **비용 최적화**: 불필요한 `Low` 이슈 생성을 억제하여 LLM 토큰 사용량 절감.

## 2. 이중 필터링 메커니즘

### Phase 1: 프롬프트 레벨 필터링 (생성 원천 억제)
각 언어별 분석기 프롬프트(`*_analyzer.txt`)에 명시적인 페널티 지침을 추가했습니다.
- **적용 위치**: `mider/config/prompts/` 내 모든 분석 프롬프트
- **지침 내용**: `"Low severity 이슈는 절대 보고하지 마세요. 오직 Medium 이상의 실질적인 위험 요소만 추출합니다."`
- **효과**: LLM이 처음부터 `Low` 이슈를 생성하지 않도록 유도하여 응답 속도를 높이고 토큰 비용을 절감합니다.

### Phase 2: 코드 레벨 필터링 (최종 방어선)
LLM의 비결정성(Non-determinism)으로 인해 지침을 어기고 `Low` 이슈를 생성하는 경우를 대비하여 Python 코드에서 강제로 필터링합니다.
- **적용 위치**: `mider/agents/` 내 각 Analyzer(ProC, C, JS, XML, SQL)의 `run()` 메서드
- **로직**:
  ```python
  # 최종 이슈 리스트를 AnalysisResult로 반환하기 직전 필터링
  filtered_issues = [
      issue for issue in issues 
      if issue.get("severity", "low").lower() != "low"
  ]
  ```
- **효과**: 어떠한 경우에도 사용자 화면과 리포트에 `Low` 등급이 노출되지 않도록 기술적으로 보장합니다.

## 3. 적용 사례 분석 (test_zinvboth08000.pc)
실제 테스트 과정에서 이중 필터링이 작동한 사례입니다.
1. **프롬프트 지침에도 불구하고**: LLM이 분석 결과로 3개의 이슈를 생성함.
2. **코드 레벨 필터 작동**: 3개 중 `severity: "low"`인 1건을 감지하여 즉시 제거.
3. **최종 출력**: 사용자에게는 검증된 2개의 `Medium/High` 이슈만 전달됨.
   - *로그 기록: `최종: 2개 이슈 (LLM 전: 2, LLM 후: 3, 병합 후: 2)`*

## 4. 기대 효과
1. **사용자 신뢰도**: 리포트에 나오는 모든 지적 사항이 '진짜 장애 위험'이라는 인식을 심어줌.
2. **운영 효율**: 수정이 불필요한 사소한 이슈를 검토하는 시간을 제거.
3. **시스템 성능**: LLM 출력 텍스트 양이 평균 10~20% 감소하여 전체 분석 시간 단축.

## 5. 향후 관리
향후 특정 이슈의 등급을 조정하거나, 다시 `Low` 등급을 보고 싶을 경우 프롬프트의 지침을 수정하고 에이전트 코드의 필터링 조건만 변경함으로써 유연하게 대응할 수 있습니다.
