## 분석 절차 (Chain-of-Thought)

### Step 1: ESLint 에러 분석
각 ESLint 에러에 대해:
1. 에러가 발생한 코드 라인을 확인합니다
2. 에러의 실제 영향도를 판단합니다 (false positive 가능성 포함)
3. 수정 방안을 제시합니다

### Step 2: 심층 패턴 분석
ESLint가 감지하지 못하는 아래 패턴을 추가로 검사합니다:

**메모리 누수 (memory_safety)**
- addEventListener 후 removeEventListener 누락
- setInterval/setTimeout 후 clearInterval/clearTimeout 누락
- DOM 참조를 변수에 저장 후 해제하지 않음

**보안 취약점 (security)**
- innerHTML, outerHTML에 사용자 입력 직접 대입
- dangerouslySetInnerHTML 사용
- eval(), Function() 생성자 사용
- document.write() 사용

**NULL/undefined 안전성 (null_safety)**
- optional chaining (?.) 없이 중첩 객체 접근
- 함수 반환값 null 체크 누락

**에러 처리 (error_handling)**
- try-catch에서 빈 catch 블록
- Promise .catch() 누락
- async 함수에서 try-catch 누락

**성능 (performance)**
- 루프 내 DOM 접근
- N+1 패턴 (루프 내 API/DB 호출)

### Step 3: 이슈 정리
- source가 "static_analysis"인 이슈: ESLint가 탐지한 것
- source가 "llm"인 이슈: LLM이 추가로 발견한 것
- source가 "hybrid"인 이슈: ESLint 결과를 LLM이 재분석한 것

## 이슈 예시
{{
  "issue_id": "JS-001",
  "category": "security",
  "severity": "critical",
  "title": "innerHTML에 사용자 입력 직접 대입으로 XSS 취약점 발생",
  "description": "사용자 입력값을 검증 없이 innerHTML에 직접 대입하고 있습니다. 공격자가 악성 스크립트를 삽입할 수 있어 XSS 공격에 취약합니다.",
  "location": {{
    "file": "/app/ui/order.js",
    "line_start": 45,
    "line_end": 45,
    "column_start": 5,
    "column_end": 40
  }},
  "fix": {{
    "before": "element.innerHTML = userInput;",
    "after": "element.textContent = userInput;",
    "description": "innerHTML 대신 textContent를 사용하여 HTML 파싱을 방지합니다."
  }},
  "source": "llm",
  "static_tool": null,
  "static_rule": null
}}

## 주의사항
- 이슈 ID는 JS-001부터 순차적으로 부여합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다 (전체 파일 Diff 금지).
- 발견된 이슈가 없으면 빈 배열 `"issues": []`로 응답합니다.
- severity 기준:
  - critical: 서비스 장애 또는 보안 사고 직결 (XSS, 메모리 누수 등)
  - high: 특정 조건에서 장애 발생 가능
  - medium: 코드 품질 저하, 잠재적 버그
  - low: 스타일, 가독성 개선 사항
