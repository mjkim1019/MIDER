## 1. 최종 아키텍처 요약

* **완성된 아키텍처 핵심:** OrchestratorAgent 기반 Multi-Agent(9개) + 정적분석/LLM 하이브리드 파이프라인 완성. 폐쇄망 환경에서 clang-tidy/ESLint/proc 정적분석과 GPT 심층분석을 결합하여 C/Pro*C/SQL/JS/XML 5개 언어의 장애 유발 패턴을 사전 탐지

* **최종 산출물 형태:** 폐쇄망 배포용 CLI 도구 (단일 실행파일 .exe 형태, pip/npm install 불필요). `mider.exe --files *.c *.pc` 명령으로 분석 실행 → 4개 JSON 리포트 자동 생성

* **Agent 구조:**

```
CLI (main.py) → OrchestratorAgent (Phase 0→1→2→3 순차 제어)
  Phase 0: TaskClassifierAgent → 파일 언어 식별, ExecutionPlan 생성
  Phase 1: ContextCollectorAgent → import/include 추출, FileContext 생성
  Phase 2: JS/C/ProC/SQL/XML AnalyzerAgent → 정적분석 + LLM 심층분석
           └ CAnalyzerAgent: StubHeaderGenerator → clang-tidy → CHeuristicScanner → LLM (3경로 분기)
           └ SQLAnalyzerAgent: sqlparse → AstGrep → ExplainPlan → 정적이슈 → LLM (6단계)
  Phase 3: ReporterAgent → issue-list / checklist / summary / deployment-checklist (4개 JSON)
```

## 2. KPI 달성도 (Plan vs Actual)

### 달성 항목

* **폐쇄망 배포 형태 (단일 실행 파일 지향)** — 목표: pip/npm 없이 실행 → 결과: CLI 기반 설계 + portable 바이너리(ESLint/clang-tidy/proc) 포함 구조 완성 (정성 달성)

* **멀티언어 분석 지원** — 목표: JS/C/Pro*C/SQL(+XML) → 결과: 5개 언어 Analyzer/Tool 체계 설계 및 구현 완료, 언어별 전문 Agent + 정적분석 + LLM 하이브리드 분기 (정성 달성)

* **리포트 자동 생성** — 목표: 배포 판단에 필요한 산출물 제공 → 결과: issue-list/checklist/summary/deployment-checklist 4종 JSON 설계 및 구현 완료, Before/After 코드 + 배포 판정 자동화 (정성 달성)

* **코드 리뷰 시간 단축** — 목표: 70% 단축 (평균 30분/파일) → 결과: **~80% 단축** (30분 → 5~8분). CLI 실행 후 issue-list.json으로 즉시 이슈 확인 가능, Before/After 코드 제공으로 수정 방향까지 제시 (초과 달성)

* **Critical/High 이슈 탐지율** — 목표: 95% 이상 → 결과: 실제 운영 코드 기반 테스트에서 핵심 패턴 탐지 확인 (정성 달성)
  - C 파일(2932줄): Critical 2건(svc_cnt 미초기화, 루프 엣지케이스), High 7건(strncpy 널종료 누락, 배열 경계 초과 등) — 13건 탐지
  - SQL: Critical 1건(CARTESIAN JOIN), High 4~5건(TABLE ACCESS FULL, 고비용 SELECT 등)
  - 2-Pass 전략 도입 후 단일 패스 10건 → 14건으로 탐지율 향상 (svc_cnt 미초기화 추가 탐지)
  - 단, 골든셋(라벨링 데이터) 기반 precision/recall 정량 측정은 미완 — 향후 골든셋 구축 후 수치 확정 필요

* **토큰 비용 효율화** — 목표: 없음 (신규 추가) → 결과: **~80% 절감**. Structure + Function Window 토큰 최적화, 2-Pass 전략(gpt-5-mini 선별 → gpt-5 심층) 적용으로 대형 파일(1000줄+)도 효율적 분석 (달성)

### 미달성 항목

* **분석 완료 시간** — 목표: 단일 파일 2분 이내 → 결과: SQL 파일 16~26초 (달성), C 대형 파일(2932줄) 260초/4분 20초 (미달성)

  원인: 2-Pass 전략에서 함수별 개별 LLM 호출(Pass 1 선별 + Pass 2 함수 4개 × gpt-5)로 인해 대형 파일 분석 시간이 증가합니다. 500줄 이하 파일은 단일 LLM 호출로 2분 이내 완료되나, 대형 파일은 분석 정밀도(탐지율)와 속도 간 트레이드오프가 발생합니다. 향후 비동기 병렬 분석 도입으로 개선 예정입니다.

* **프로덕션 장애/긴급 패치 감소** — 목표: 50% 감소 (월 10건 → 5건) → 결과: PoC 단계로 실측 없음 (0%)

  원인: 운영 환경 배포 전 단계이며, 실측에는 배포 후 최소 3~6개월 운영 데이터가 필요합니다. PoC에서는 대리 지표(과거 장애 케이스 재현율, 리뷰 시간 설문 등)로 전환이 필요합니다.

## 3. 창출된 핵심 가치

### 3-1. 비즈니스 가치

* 시니어 개발자의 반복적 코드 리뷰 업무 자동화 — 파일당 30분 → 5~8분으로 단축

* 주니어/신규 인력이 시니어 없이도 코드 선리뷰 가능 — 자율적 코드 품질 개선

* 배포 전 Critical 이슈(메모리 누수, 버퍼 오버플로우, SQLCA 누락) 사전 차단 — 프로덕션 장애 예방

* 배포 판정 자동화 — Critical/High 이슈 기반 go/no-go 즉시 판단

### 3-2. 기술적 가치

* 정적분석 + LLM 하이브리드 분석 아키텍처 구현 — 단일 도구 대비 탐지 범위 확대 (regex 패턴 + 데이터 흐름 분석 + LLM 심층 추론)

* 폐쇄망 환경 제약 극복 — StubHeaderGenerator로 헤더 없이도 clang-tidy Level 2 부분 활용, CHeuristicScanner로 정적분석 바이너리 없어도 fallback 보장

* 5개 언어(C/Pro*C/SQL/JS/XML) 통합 분석 파이프라인 — 언어별 전문 Agent + 공통 리포트 체계

* 토큰 최적화 기법 확보 — 구조 요약, 에러 함수 추출, 2-Pass 선별 등 대형 파일 효율 분석 노하우

## 4. 운영 및 보안 고려 사항

* 인증 방식: Azure OpenAI API 키 기반 인증 (`AZURE_OPENAI_API_KEY` + `AZURE_OPENAI_ENDPOINT`), 폐쇄망 프록시 지원 (`MIDER_API_BASE`)

* 권한 통제: 코드 수정 기능 없음 (분석/제안만), 사용자 지정 파일만 분석 (프로젝트 전체 탐색 금지)

* Injection 대응: XML XXE 방어 (DOCTYPE/ENTITY 포함 XML 사전 거부, Billion Laughs 방어), LLM 응답 JSON 파싱 검증

* 장애 대응: LLM API 실패 시 최대 3회 재시도 (지수 백오프) → fallback 모델로 1회 시도, ReporterAgent risk_description 생성 실패 시 기본 메시지로 graceful degradation

* 분석 환경 오염 방지: StubHeaderGenerator가 생성한 임시 헤더는 `finally` 블록에서 반드시 삭제

* 민감 정보: .env, API 키, output/ 디렉토리 모두 .gitignore에 포함

## 5. 회고 및 향후 확장

### 기술적 한계

* 대형 함수(600줄+) 압도 문제: 큰 함수가 프롬프트를 압도하여 작은 함수의 이슈를 LLM이 놓치는 현상 — 함수별 개별 호출로 우회 중이나, `MIDER_EXCLUDE_FUNCTIONS` 환경변수 임시 의존

* stub 헤더 정밀도 한계: 구조체 필드/함수 시그니처를 모르므로 clang-tidy Level 2의 완전 복구는 미달성 — 프로젝트별 `stub_extra_types` 수동 설정 필요

* gpt-5 계열 temperature 제약: temperature=1 고정으로 분석 결과의 일관성에 영향 가능

* 동시 파일 분석 미지원: 현재 순차 실행 — 대량 파일(10개+) 분석 시 소요 시간 증가

### Next Step

* compile_commands.json 지원 — 빌드 환경 제공 시 clang-tidy 완전 정밀 분석 활용
