## 분석 절차 (Chain-of-Thought)

### Step 1: 함수별 경고 전수 분석 (필수)
**위 정적분석 결과는 함수별로 그룹화되어 있습니다. 모든 함수의 경고를 빠짐없이 분석하세요.**

분석 순서:
1. **각 함수 섹션을 순서대로** 분석합니다 (첫 번째 함수만 분석하고 끝내지 마세요)
2. 각 함수에서 UNINIT_VAR, UNSAFE_FUNC, MALLOC_NO_CHECK 경고는 **반드시** 이슈로 보고합니다
3. UNINIT_VAR: 해당 변수가 이후 어디서 사용되는지 추적하여 장애 가능성을 설명합니다
4. 함수가 4개이면 4개 함수 모두에서 이슈를 찾아야 합니다

### Step 2: 심층 패턴 분석
clang-tidy가 감지하지 못하는 아래 패턴을 추가로 검사합니다:

**버퍼 오버플로우 (memory_safety)**
- strcpy, sprintf 사용 (strncpy, snprintf 대신)
- 배열 크기를 넘는 인덱스 접근
- gets() 사용 (fgets 대신)

**NULL 포인터 역참조 (null_safety)**
- malloc/calloc 반환값 NULL 체크 누락
- 함수 반환 포인터 NULL 체크 누락
- 포인터 매개변수 NULL 검증 누락

**메모리 누수 (memory_safety)**
- malloc 후 free 누락 (특히 에러 경로)
- 함수 중간 return 시 할당된 메모리 미해제
- realloc 실패 시 원본 포인터 유실

**Use-After-Free / Double-Free (memory_safety)**
- free 후 포인터 재사용
- 같은 포인터를 두 번 free
- free 후 포인터를 NULL로 설정하지 않음

**스레드 안전성 (performance)**
- 전역 변수 접근 시 mutex 미사용
- 공유 자원 접근 시 동기화 누락

### Step 3: 이슈 정리
- source가 "static_analysis"인 이슈: clang-tidy가 탐지한 것
- source가 "llm"인 이슈: LLM이 추가로 발견한 것
- source가 "hybrid"인 이슈: clang-tidy 결과를 LLM이 재분석한 것

  "issue_id": "C-001",

## 주의사항
- 이슈 ID는 C-001부터 순차적으로 부여합니다.
- title과 description은 반드시 한국어로 작성합니다.
- fix의 before/after는 1-3줄만 포함합니다.
- 발견된 이슈가 없으면 `"issues": []`로 응답합니다.
- C 언어의 메모리 안전성 이슈는 severity를 높게 책정합니다 (critical 또는 high).
- **정적분석 결과의 모든 경고를 누락 없이 이슈로 변환하세요. 경고를 건너뛰지 마세요.**
- **[제외] 구조체/변수 초기화 선언 자체를 이슈로 보고하지 마세요.** C/Pro*C에서 `struct xxx_io; memset(&xxx_io, 0, sizeof(xxx_io));` 패턴은 표준 관행입니다. 선언 후 memset/DB호출로 채우는 경우는 정상입니다. 초기화 없이 **실제 사용(읽기/인덱싱)** 하는 경우만 이슈로 보고하세요.
- **동일 패턴 반복은 대표 1건으로 병합하세요.** 예: svc_cnt 경계 미검사 인덱싱이 10곳이면 대표 1건 + "외 N곳 동일 패턴"으로 보고합니다.
- **[중요] fix.before/after에는 원본 코드에 실제로 존재하는 변수명과 코드만 사용하세요.** 원본에 없는 변수명을 만들거나, 변수명을 임의로 변경하지 마세요. 예: 원본이 `curcnt`이면 after에서도 `curcnt`를 사용해야 합니다. `n`, `cnt`, `count` 등으로 바꾸지 마세요.
